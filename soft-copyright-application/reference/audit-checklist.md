# 软著申请材料审计检查清单

本清单来源于实际软著申请材料的审计意见（round-1、round-2），编写或修订材料时必须逐项满足。

---

## Round-1：敏感词与合规表述

**依据**：2021 年 9 月十部委《关于进一步防范和处置虚拟货币交易炒作风险的通知》等，虚拟货币相关业务活动属于非法金融活动。

**敏感词列表（需避免或替换为中性技术表述）**：

- 虚拟货币
- 虚拟资产
- 数字收藏
- 数字藏品
- 数字代币
- NFT
- token（语境若涉及金融/投机需弱化或替换）
- 数字钱包

**处理建议**：可将「代币」等替换为「token」等中性技术用语；整体表述侧重技术能力（如链上资产标识、接口），避免理财、投机、金融属性。

---

## Round-2：软件全称与格式

1. **软件全称不得有多余空格**  
   - 错误示例：`国际区块链 Python SDK 软件 V1.0`（若作为全称含多余空格或版本号混入）。  
   - 正确示例：`国际区块链 Python SDK 软件`（全称中无多余空格，版本号在申请表单独填写）。

2. **软件全称须以「XX 系统 / XX 软件 / XX 平台」结尾**  
   - 与申请表、说明书、截图中的名称保持一致。

---

## Round-2：源码前后 1500 行起止

1. **《前 1500 行》**  
   - 前几页必须是申请软件源码**某个模块的开始代码**；  
   - 第 1 行应为该模块的起始有效代码（与说明书叙述顺序一致）。

2. **《后 1500 行》**  
   - 后几页必须是**某个模块的结束代码**；  
   - 一般按**说明书功能说明的顺序**，**最后一个功能**的**最末尾代码**作为整份「后 1500 行」的最后一页/最后一行。  
   - 即：后半代码文件应包含最后一个功能，并以该功能的最后一行代码结束。

**操作要点**：先定说明书功能顺序与最后一节对应源码位置（文件 + 行号），再逆推「后 1500 行」的文件顺序与截取区间（可参考 `BACK_FILE_ORDER`、`BACK_END_AT` 的实现）。

---

## Round-2：说明书模块化与内容深度

1. **功能模块化**  
   - 按功能划分章节，每个功能独立成节。

2. **每节必须包含**  
   - 功能概述（文字说明）；  
   - **流程图**（建议 Mermaid）；  
   - **体系结构**（模块/层次、依赖关系）；  
   - **数据结构**（输入/输出、关键类型）；  
   - **接口**（如接口表：方法名、参数、说明）；  
   - **构件**（若适用）；  
   - **算法或处理逻辑**的清晰表达。

3. **深度要求**  
   - 将「框架」细化为**具体数据结构**和**软件算法**的详细表达与介绍说明，避免仅停留在高层描述。

---

## 终审自查项（提交前必查）

- [ ] 敏感词已按 round-1 处理；
- [ ] 软件全称无多余空格，且以「系统/软件/平台」结尾；
- [ ] 《前 1500 行》以某模块**开始**代码起头；
- [ ] 《后 1500 行》以说明书**最后功能**的**最末代码**收尾；
- [ ] 说明书每个功能含：流程图、体系结构、数据结构、接口、算法说明；
- [ ] 申请表、说明书、源码量、软硬件环境、软件全称/简称在各处一致；
- [ ] 截图与全称/简称一致，截图环境与申请表一致；无他方名称/域名（或已附授权/ICP 说明）；
- [ ] 代码中无 `Copyright + 其他公司名称`（本软件名可保留）。
